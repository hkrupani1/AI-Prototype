## Top Tips for PMs Using AI

**1. Be Specific with Context**
Always include your product name, target persona, stage, and constraints in your prompts. Vague prompts = vague output.

**2. Use Role Priming**
Start prompts with *"Act as a senior product manager with expertise in [domain]..."* to improve quality of output.

**3. Iterate, Don't Accept First Draft**
Use follow-up prompts like *"Make it more concise"*, *"Add a risk section"*, or *"Reformat as a table"*.

**4. Chain Workflows**
Use output from one AI task as input to the next (e.g., use persona output → feed into user story generation → feed into acceptance criteria).

**5. Validate AI Output**
AI can hallucinate facts especially for market data. Always verify with authoritative sources before sharing with stakeholders.

**6. Build a Prompt Library**
Save your best-performing prompts in Notion or a shared doc so your entire PM team can reuse them consistently.

### 1. Publish High-Level PRD (User/Business Need, Intent, Mocks)

**Current Tools:** Word, OneNote  
**AI Tools:** Copilot for Word, M365 Copilot, GitHub Copilot

#### How to Do It With AI

**Step 1 — Gather source material first**
Before writing a single word, use M365 Copilot to pull together all relevant context:
- In **Copilot Chat (Teams/M365)**: *"Summarize all emails and Teams messages about [feature name] from the last 3 months. Focus on: customer pain points, business goals, and any decisions made."*
- Attach meeting transcripts, customer feedback exports, or research docs as context.

**Step 2 — Auto-draft the PRD structure in Word**
Open a new Word doc and use **Copilot in Word**:
> *"Draft a high-level PRD for [feature name]. Use this context: [paste summary]. Include: Executive Summary, Problem Statement, User Personas, Business Need, Goals & Success Metrics, High-Level Solution Intent, and Out of Scope."*

**Step 3 — Generate the User/Business Need narrative**
> *"Write a 2-paragraph user need statement for [persona] who struggles with [pain point]. Format it as: Current State → Pain → Desired Future State → Business Impact."*

**Step 4 — Describe mocks in the PRD (before Figma is ready)**
> *"Describe the key UI screens needed for [feature] as a written mock narrative. For each screen include: screen name, purpose, key UI elements, user action, and system response."*

**Step 5 — Iterate and refine**
- *"Make the executive summary more concise — 3 sentences max."*
- *"Rewrite the goals section to be more measurable with specific KPIs."*
- *"Add a risks and assumptions section."*

#### Key Tips
- Use `create-brd.prompt.md` from the repo (GitHub Copilot) for structured BRD auto-generation from M365 sources
- Ground every section in real data — paste actual customer quotes, support ticket themes, or NPS comments into the prompt
- Never start from a blank page; always prime Copilot with source material first

### 2. Identify Dependencies (Internal & External) and Get Commitment

**Current Tools:** Email, Meetings  
**AI Tools:** Copilot in ADO, GitHub Copilot + ADO MCP Server, Copilot in Outlook/Teams

#### How to Do It With AI

**Step 1 — Extract dependencies from specs and discussions**
Use GitHub Copilot (Agent Mode) with your spec file open:
> *"Read this spec and identify all internal and external dependencies. For each dependency list: team/system name, what they need to provide, and the likely blocking date."*

Or from meeting transcripts via `DiscussionToADOTasksAgent.prompt.md`:
> *"Extract all dependency commitments, open questions, and action items from this meeting transcript. Create ADO work items for each."*

**Step 2 — Build a dependency tracker in ADO**
Use `ado-release-dependencies-updater.prompt.md` (GitHub Copilot + ADO MCP):
> *"Update the 'Has Work' field for release [ID] with all C+E planning dependencies from Service Tree. Show me which teams are blocking and which are blocked."*

**Step 3 — Draft commitment request emails**
Use **Copilot in Outlook**:
> *"Write a dependency commitment request email to [team name]. Context: we need [specific deliverable] by [date] to unblock [feature]. Tone: collaborative and specific. Include: what we need, why it matters, proposed deadline, and ask for explicit confirmation."*

**Step 4 — Follow-up and track**
Use **Copilot in Teams** to summarize dependency threads:
> *"Summarize the conversation in [channel/thread] about the [feature] dependency. Extract: current status, who has committed, what is still open, and next steps."*

**Step 5 — Validate commitment completeness before intake**
Use `intake-screening.prompt.md`:
> `/intake-screening [work-item-id]` — checks that "Has Work" / team assignments are populated before formal intake

#### Key Tips
- Create a dependency matrix table in Word using Copilot: *"Create a dependency matrix table with columns: Dependency, Owner Team, Type (Internal/External), Deliverable, Required By, Status, Risk If Late"*
- Use `exit-criteria-check.prompt.md` to verify all dependencies are formally committed before moving to next lifecycle stage
- For external dependencies, use Copilot to draft the escalation chain and timing assumptions

### 3. Review Lo-Fi Figma from Design Team

**Current Tools:** Figma, Meetings  
**AI Tools:** Copilot in Figma, GitHub Copilot (with spec context), Copilot in Teams

#### How to Do It With AI

**Step 1 — Export Figma frames as images/PDF before the review meeting**
Ask the designer to export lo-fi frames. Then use **Copilot in Teams or ChatGPT with Vision**:
> *"Review these lo-fi mockups against the PRD requirements I'll paste below. For each screen: (1) Does it satisfy the stated user need? (2) Are there missing flows or edge cases? (3) What requirements are not yet represented?"*

**Step 2 — Generate structured review feedback**
Use GitHub Copilot with your PRD/spec open:
> *"Based on the requirements in this PRD [attach/paste], generate a design review checklist for evaluating lo-fi mockups. Include: user flow completeness, error states, accessibility considerations, edge cases, and mobile/responsive checks."*

**Step 3 — Summarize design review meeting**
Use `meeting-notes-formatter.prompt.md` after the Figma review call:
> *"Format these raw Teams meeting notes into a structured design review summary. Include: decisions made, open design questions, feedback items by screen, and next steps with owners."*

**Step 4 — Cross-reference design with acceptance criteria**
> *"Compare these Figma screen descriptions [paste] against the acceptance criteria in this PRD [paste]. Identify: (a) screens that satisfy criteria, (b) criteria not yet covered, (c) screens shown that are out of scope."*

**Step 5 — Create ADO feedback tasks from design review**
Use `DiscussionToADOTasksAgent.prompt.md`:
> *"From this design review feedback list [paste], create ADO tasks for the design team. For each: title, description, priority, linked user story, and acceptance condition."*

#### Key Tips
- Use **Copilot in Figma** (if enabled) to auto-check designs against design system components and accessibility rules
- Always bring your PRD acceptance criteria into the review as the source of truth — have Copilot map each screen to a requirement
- Ask Copilot: *"What user journeys are NOT covered by these mockups?"* to find gaps before dev starts

### 4. Publish Detailed PRD with Acceptance Criteria

**Current Tools:** Word, ADO  
**AI Tools:** Copilot for Word, GitHub Copilot (spec generation), `create-tech-spec.prompt.md`

#### How to Do It With AI

**Step 1 — Upgrade high-level PRD to detailed spec**
Use `convert-file-to-spec.prompt.md` in GitHub Copilot:
> *"Convert this high-level PRD [attach] into a detailed C+E Specification. Add: detailed functional requirements, non-functional requirements, data model changes, API impacts, error handling, and security considerations."*

Or use Copilot for Word:
> *"Expand this PRD section by section into a detailed specification. For each requirement, add: detailed description, acceptance criteria, edge cases, and assumptions."*

**Step 2 — Generate Gherkin-style acceptance criteria**
For each feature/scenario in your spec:
> *"Write Gherkin-style acceptance criteria (Given-When-Then) for this requirement: [paste requirement]. Cover: happy path, error states, boundary conditions, and permission/role variants."*

**Step 3 — Review spec quality with AI**
Use `Commerce-Feature-Spec-Companion.prompt.md`:
> *Set `target_spec` to your spec file, `mode` = "improve"*
> The agent reviews your spec against 31 real production specs and gives concrete suggestions with examples.

Use `Commerce-Spec-Full-Review.prompt.md` for the full review:
> Phase 1 scores spec quality → Phase 2 maps component impact → unified readiness report

**Step 4 — Score the BRD and update ADO**
Use `score-brd.prompt.md`:
> `/score-brd [ADO-work-item-id]` — AI scores against 4 dimensions and auto-populates ADO fields with scores + justifications

**Step 5 — Validate ADO spec links and completeness**
Use `intake-screening.prompt.md`:
> `/intake-screening [work-item-id]` — verifies BRD/Spec URL is present, description is complete, and scoring is done

#### Key Tips
- Use the **BRD Evaluator** pipeline to get an automated quality score before stakeholder distribution
- For each acceptance criterion, ask Copilot: *"What test cases would verify this criterion passes?"* to pre-seed QA planning
- Use `mermaid-helper.prompt.md` to generate sequence diagrams for complex scenarios in the spec
